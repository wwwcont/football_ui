# Product + UX Specification: Веб-приложение футбольного турнира

## 0) Назначение документа

Документ фиксирует функциональные и UX-требования к production-ready системе управления футбольным турниром с двумя контурами:

- **Public UI** (публичная часть для зрителей, игроков, представителей команд, медиа).
- **Admin UI** (операционный контур для ведения турнира в реальном времени).

Спецификация предназначена одновременно для команд **frontend** и **backend** как единое ТЗ по доменной модели, ролям, жизненному циклу, экранам, потокам и обработке пограничных сценариев.

---

## 1) Цели продукта и продуктовые ограничения

## 1.1 Цели

1. Обеспечить прозрачный и актуальный доступ к результатам турнира в public-интерфейсе.
2. Дать админской команде инструменты для надежного и быстрого операционного ведения матчей.
3. Поддержать полный цикл сезона: от регистрации команд до финальной статистики и архива.
4. Минимизировать ручные ошибки за счет валидируемых workflows, прав ролей и автоматических пересчетов.

## 1.2 Нефункциональные ограничения

- Интерфейс: **mobile-first**, адаптивный до desktop.
- Визуальный стиль: **черно-белый минимализм**, акцент на типографику и контраст, минимум визуального шума.
- Логотипы команд: загрузка и хранение в **PNG**.
- Система рассчитана на **реальную эксплуатацию**:
  - одновременная работа нескольких операторов/администраторов;
  - журналирование действий;
  - предсказуемая консистентность статистики;
  - устойчивость к сетевым сбоям и повторным отправкам.

## 1.3 Ключевые KPI (ориентиры)

- Time-to-publish результата матча после финального свистка: ≤ 2 мин.
- Доля матчей без последующих ручных корректировок: ≥ 95%.
- Ошибки прав доступа (попытка запрещенного действия): 0 критических инцидентов.
- Доступность public-части в дни матчей: 99.9%.

## 2) Роли пользователей и права

## 2.1 Viewer

**Кто:** любой неавторизованный или авторизованный пользователь public-части.

**Права:**
- Просмотр календаря, матчей, таблиц, статистики, новостей.
- Поиск команд/игроков/матчей.
- Подписка на уведомления (если реализуется).

**Ограничения:**
- Нет прав на изменение данных.

## 2.2 Captain / Team Manager

**Кто:** представитель конкретной команды.

**Права:**
- Просмотр и редактирование карточки своей команды (ограниченный набор полей).
- Управление ростером команды в рамках регламента (добавить/удалить игрока, номера, позиции, документы при наличии).
- Предматчевое формирование заявки на матч (match squad) своей команды.
- Подтверждение протокола/результата со стороны команды (по правилам лиги).

**Ограничения:**
- Доступ только к своей команде/своим заявкам.
- Не может напрямую менять счет, события матча, турнирные правила.

## 2.3 Match Operator

**Кто:** оператор матча (секретарь/статист).

**Права:**
- Запуск лайв-ведения матча.
- Ведение таймлайна: голы, карточки, замены, автоголы, пенальти, отмененные события (с reason code).
- Фиксация статуса матча (first half, halftime, second half, extra time, finished, interrupted и т.д.).
- Черновая и финальная фиксация результата с отправкой на подтверждение.

**Ограничения:**
- Работает только с назначенными матчами/дивизионом.
- Не меняет структуру сезона, состав дивизионов, глобальные настройки.

## 2.4 League Admin

**Кто:** администратор турнира/лиги.

**Права:**
- Создание/редактирование сезонов, дивизионов, туров, матчей.
- Управление командами и игроками (в рамках своей лиги).
- Публикация расписания.
- Подтверждение спорных результатов, ручные корректировки (с audit log).
- Публикация новостей/событий.
- Назначение операторов на матчи.

**Ограничения:**
- Ограничение scope-ом своей лиги/организации.
- Не управляет глобальными системными настройками и ролями super admin.

## 2.5 Super Admin

**Кто:** системный администратор платформы.

**Права:**
- Полный доступ ко всем лигам и сезонам.
- Управление ролями/правами admin users.
- Модерация/блокировка сущностей, восстановление после ошибок.
- Управление справочниками, глобальными параметрами, системными интеграциями.
- Доступ к расширенному аудиту и административным операциям.

**Ограничения:**
- Операции высокого риска требуют подтверждения (2-step confirm) и протоколируются.

## 3) Ключевые сущности (доменная модель)

Ниже — минимальные поля и связи. Формат IDs: UUID.

## 3.1 Season

**Назначение:** временной контур турнира.

**Поля:**
- id
- name (например, "Spring League 2026")
- status: draft | registration | scheduled | live | completed | archived
- start_date, end_date
- ruleset_id (ссылка на набор правил подсчета)
- visibility: private | public
- created_at, updated_at, created_by

**Связи:**
- 1:N с division/group
- 1:N с news/event

## 3.2 Division / Group

**Назначение:** подтурнир/группа внутри сезона.

**Поля:**
- id
- season_id
- name
- format: round_robin | double_round_robin | knockout | mixed
- sort_order
- status

**Связи:**
- M:N с team (через membership)
- 1:N с match
- 1:1/1:N с standings snapshot

## 3.3 Team

**Назначение:** клуб/команда-участник.

**Поля:**
- id
- name_short, name_full
- logo_upload_id (PNG)
- captain_user_id (optional)
- manager_user_id (optional)
- contact_phone, contact_email
- status: active | disqualified | withdrawn
- metadata (цвет формы, город и пр.)

**Связи:**
- 1:N с player
- M:N с division/group
- 1:N с match (home/away)
- 1:N с team stats

## 3.4 Player

**Назначение:** игрок команды.

**Поля:**
- id
- team_id
- first_name, last_name
- birth_date
- number
- position
- is_captain
- status: active | injured | suspended | inactive
- optional docs/eligibility flags

**Связи:**
- M:N с match через match squad
- 1:N с match event
- 1:N с player stats

## 3.5 Match

**Назначение:** матч календаря.

**Поля:**
- id
- season_id, division_id, round_id(optional)
- home_team_id, away_team_id
- venue
- kickoff_at (timezone-aware)
- status: scheduled | live | finished | postponed | canceled | technical_result
- score_home, score_away
- result_type: regular | extra_time | penalties | technical
- operator_user_id
- is_published
- locked_at (после финального подтверждения)

**Связи:**
- 1:N с match squad
- 1:N с match event

## 3.6 Match Squad

**Назначение:** заявка команды на конкретный матч.

**Поля:**
- id
- match_id, team_id
- player_id
- role: starter | substitute
- shirt_number_snapshot
- is_confirmed_by_team
- confirmed_at

**Особенность:** снапшотные поля, чтобы исторические данные не ломались при изменении профиля игрока.

## 3.7 Match Event

**Назначение:** событие матча в хронологии.

**Поля:**
- id
- match_id
- team_id
- player_id (nullable для team-level события)
- event_type: goal | own_goal | yellow_card | red_card | substitution | penalty_scored | penalty_missed | var_cancelled | note
- minute, extra_minute
- period: 1H | HT | 2H | ET1 | ET2 | PEN
- related_player_id (для substitution)
- status: active | canceled
- canceled_reason
- created_by, created_at

## 3.8 Standings

**Назначение:** турнирная таблица по дивизиону.

**Поля (snapshot/derived):**
- id
- season_id, division_id
- team_id
- played, won, draw, lost
- goals_for, goals_against, goal_diff
- points
- fair_play_points (optional)
- rank
- tie_break_explain (техническое поле)
- calculated_at

## 3.9 Player Stats

**Назначение:** агрегированная статистика игрока.

**Поля (derived + optional manual):**
- player_id, season_id, division_id
- matches_played
- goals
- assists (если ведется)
- yellow_cards, red_cards
- clean_sheets (для GK, если ведется)
- minutes_played (optional)
- updated_at

## 3.10 Team Stats

**Назначение:** агрегированная статистика команды.

**Поля:**
- team_id, season_id, division_id
- matches_played
- wins/draws/losses
- goals_for/goals_against
- clean_sheets
- form_last_5
- updated_at

## 3.11 Admin User

**Назначение:** внутренний пользователь админ-контура.

**Поля:**
- id
- email, phone
- full_name
- role: match_operator | league_admin | super_admin
- scopes (league_ids/division_ids)
- status: active | suspended
- last_login_at

## 3.12 News / Event

**Назначение:** информационные публикации для public UI.

**Поля:**
- id
- season_id(optional), division_id(optional)
- title
- summary
- body (rich text, ограниченный sanitizer)
- cover_upload_id(optional)
- published_at
- status: draft | scheduled | published | archived
- author_user_id

## 3.13 Upload

**Назначение:** единый реестр файлов.

**Поля:**
- id
- file_type: team_logo_png | news_image | document
- mime_type
- size_bytes
- storage_key
- width, height (для изображений)
- checksum
- uploaded_by
- uploaded_at
- status: active | deleted

**Правила для team logo:**
- Только PNG.
- Валидация размера файла и минимального разрешения.
- Автогенерация превью/thumbnail.

## 4) Жизненный цикл турнира (end-to-end)

## Этап 1. Создание сезона
- League Admin создает Season в статусе `draft`.
- Настраиваются даты, формат, базовые правила подсчета.
- При необходимости создаются дивизионы/группы.

## Этап 2. Создание команд
- Добавляются Team с базовыми данными.
- Загружается логотип (PNG).
- Назначается captain/team manager (если пользователь уже существует).

## Этап 3. Добавление игроков
- Команды/админ наполняют ростеры.
- Проверяется уникальность номера в рамках команды (по регламенту).
- Игроки переводятся в `active` после валидации.

## Этап 4. Создание туров
- Генерируются раунды (авто или вручную).
- Проверяются конфликтные слоты (поле/время/команда).

## Этап 5. Создание матчей
- Для каждой пары формируется Match.
- Назначаются дата/время/поле/оператор.
- Матчи в `scheduled`.

## Этап 6. Публикация расписания
- League Admin публикует расписание (флаг `is_published`).
- Public UI показывает календарь и карточки предстоящих матчей.

## Этап 7. Ведение матча
- Перед матчем: команды подтверждают match squads.
- Во время матча: оператор фиксирует события и статусные переходы.
- Изменения должны быть near real-time в public UI (через polling/WebSocket/SSE).

## Этап 8. Подтверждение результата
- По окончании оператор отправляет финальный протокол.
- При необходимости требуется подтверждение League Admin и/или капитанов.
- После подтверждения Match блокируется (`locked_at`), доступен только flow корректировки с audit trail.

## Этап 9. Пересчет таблиц и статистики
- Триггер пересчета standings, player stats, team stats.
- Пересчет идемпотентный и детерминированный.
- Ошибки пересчета уходят в очередь повторов + алерты админам.

## Этап 10. Публикация данных в Public UI
- Обновленные таблицы, статистика, карточки матчей, новостная лента.
- В конце сезона: перевод в `completed` и затем `archived`.

## 7) Основные user flows

## 7.1 Viewer: просмотр live-матча
1. Открывает Home.
2. Переходит в live матч.
3. Видит обновляющийся таймлайн и счет.
4. После статуса `finished` получает финальный протокол и обновленную таблицу.

## 7.2 Captain/Manager: предматчевая заявка
1. Авторизация.
2. Выбор ближайшего матча своей команды.
3. Формирование стартового состава и замен.
4. Подтверждение заявки.
5. До дедлайна — разрешено редактирование, после — только через запрос админу.

## 7.3 Match Operator: ведение матча
1. Открывает назначенный матч.
2. Проверяет составы.
3. Стартует матч (`live`).
4. Вносит события по таймеру и типам.
5. Завершает матч, отправляет финальный протокол.

## 7.4 League Admin: публикация тура
1. Генерирует/редактирует тур.
2. Устраняет конфликты.
3. Публикует расписание.
4. Проверяет public отображение.

## 7.5 League Admin: спорная корректировка
1. Открывает завершенный матч.
2. Запускает flow корректировки (обязательное reason/comment).
3. Система пересчитывает standings/stats.
4. Изменение фиксируется в audit log и помечается как corrected.

## 7.6 Super Admin: управление доступами
1. Открывает Admin Users.
2. Меняет роль/scope пользователя.
3. Действие подтверждается повторно.
4. Изменение мгновенно применяется и логируется.

## 8) Mobile UX patterns

1. **Bottom navigation** (public): Home / Matches / Table / Teams / More.
2. **Sticky match header**: счет и статус всегда в зоне видимости.
3. **Progressive disclosure**: сначала ключевые данные, детали — в accordion/sheet.
4. **Thumb-friendly controls**: крупные touch targets ≥ 44px.
5. **Single-column forms** в админке, пошаговые wizard-потоки.
6. **Inline validation + clear errors** рядом с полем.
7. **Offline-aware states** для operator UI (очередь событий, retry).
8. **Skeleton loaders** вместо скачков верстки.
9. **Destructive actions** через bottom sheet confirm.

---

## 9) Desktop UX patterns

1. **Two-pane layouts**:
   - слева список матчей/сущностей,
   - справа детали/редактор.
2. **Data-dense tables** в админке с фильтрами и bulk actions.
3. **Persistent side navigation** для admin.
4. **Keyboard support**:
   - hotkeys для live-ведения (например, G=goal, Y=yellow).
5. **Split timeline + event form** на экране Match Center.
6. **Advanced filters** в виде toolbar + saved views.
7. **Audit diff view** в модальном окне/боковой панели.

## 10) Edge cases и обработка ошибок

## 10.1 Матчевые сценарии
- Техническое поражение/победа.
- Отмена матча (погода/форс-мажор).
- Прерванный матч с доигровкой.
- Дублирующиеся события (double submit).
- Исправление ошибочно внесенного гола/карточки.

## 10.2 Составы и игроки
- Игрок в заявке не состоит в команде.
- Игрок дисквалифицирован, но добавлен в squad.
- Конфликт номера игрока.

## 10.3 Расписание
- Команда назначена на два матча в один слот.
- Недоступность поля/времени.
- Перенос матча после публикации и уведомления пользователей.

## 10.4 Данные и синхронизация
- Потеря сети у оператора во время live.
- Повторная отправка одного события после reconnect.
- Конфликт одновременного редактирования двумя админами.

## 10.5 Медиа/загрузки
- Невалидный формат логотипа (не PNG).
- Слишком большой файл.
- Поврежденный PNG.

## 10.6 Безопасность
- Попытка доступа к данным вне scope роли.
- Подмена идентификаторов в запросах.
- Массовый brute-force входа в админку.

## 11) Что считается автоматически vs редактируется вручную

## 11.1 Автоматически (source of truth = match events + confirmed result)

1. Standings:
   - played/won/draw/lost
   - goals for/against, diff
   - points
   - rank по правилам tie-break

2. Player stats:
   - goals/cards/appearances и т.д. (из событий + squads)

3. Team stats:
   - форма, aggregate показатели

4. Лидерборды и витрины public UI.

5. Статусы публикации derived-виджетов после подтверждения матча.

## 11.2 Вручную (с ограничениями и аудитом)

1. Базовые данные сезона/дивизиона/команд/игроков.
2. Планирование календаря, переносы матчей.
3. Лайв-события матча (оператор вводит вручную).
4. Ручные корректировки результата/событий — только через специальный flow:
   - обязательно reason/comment,
   - кто и когда изменил,
   - авто-пересчет после сохранения.
5. News/CMS контент.

## 11.3 Полуручной режим (manual trigger + auto compute)

- Пересчет статистики по кнопке “Recalculate” при инцидентах,
- но сам расчет выполняется только серверной логикой.

## 13) Требования к API и данным (для backend)

1. Все write-операции должны быть идемпотентными (Idempotency-Key для критичных endpoint’ов).
2. Optimistic locking/versioning для сущностей, редактируемых конкурентно.
3. Аудит обязателен для всех admin write-действий.
4. Webhook/event bus для пересчета и публикации derived данных.
5. Отдельные endpoint’ы:
   - draft save,
   - submit for confirmation,
   - confirm/lock,
   - correction flow.
6. Таймзона хранения: UTC + явный timezone при отображении.
7. Историчность: матчевые данные неизменяемы без audit-протокола.

## 14) UI/Design guidelines (для frontend)

1. Черно-белая палитра:
   - фон: white/near-black,
   - текст: near-black/white,
   - акценты через плотность/границы/иконки, а не цветовой шум.
2. Компоненты:
   - card, table, list item, badge статуса, timeline item, stat chip.
3. Типографика:
   - высокий контраст,
   - читаемая иерархия (H1/H2/body/meta).
4. Состояния:
   - loading/skeleton,
   - empty,
   - error,
   - no-permission.
5. Accessibility:
   - WCAG AA контраст,
   - aria labels,
   - keyboard navigation desktop.

## 15) Операционные требования (production readiness)

1. Наблюдаемость:
   - structured logs,
   - error tracking,
   - метрики latency/error rate.
2. Резервирование:
   - регулярные backup БД,
   - восстановление на точку времени.
3. Безопасность:
   - rate limiting,
   - CSRF/XSS protection,
   - secure cookies/token rotation.
4. Надежность live-режима:
   - очередь событий при offline,
   - авто-retry,
   - дедупликация событий.
5. Поддержка масштабирования:
   - кэширование public read-model,
   - async jobs для тяжелых пересчетов.

## 16) Definition of Done (MVP релиз)

MVP считается готовым, если:

1. Реализованы все роли и ограничения прав из раздела 2.
2. Реализованы сущности и связи из раздела 3.
3. Пройден полный жизненный цикл из раздела 4 на staging.
4. Доступны обязательные public/admin экраны (разделы 5 и 6).
5. Отработаны ключевые user flows (раздел 7).
6. Поддержаны mobile и desktop UX паттерны (разделы 8 и 9).
7. Реализована обработка edge cases (раздел 10).
8. Автоматические и ручные зоны ответственности разделены (раздел 11).
9. Включен audit log и механика корректировок.
10. Система выдерживает пилотную нагрузку реального тура.
